OpenWeights Compute Architecture — Notes from 2026-03-20
========================================================

Context: My training jobs were stuck in `pending` for a day because I didn't
understand how OpenWeights provisions compute. These notes clarify the architecture.

## Two keys, two purposes

- OPENWEIGHTS_API_KEY: Authenticates you as a client. Lets you submit jobs, upload
  files, check status. Talks to the OW backend (a Supabase database). Works with
  zero workers and zero cluster managers — the database always accepts jobs.

- RUNPOD_API_KEY: Used by the Cluster Manager to provision actual GPU pods on RunPod.
  This is what pays for and creates compute. NOT stored in or retrievable from the
  OW org — it belongs to whoever starts the cluster manager.

## The Cluster Manager

The cluster manager is a separate process that bridges the OW job queue and RunPod
GPU infrastructure. It must be started manually by someone with a RUNPOD_API_KEY.

Three ways to start it:
  ow cluster --env-file .env     # Run locally
  ow deploy --env-file .env      # Deploy to a cheap RunPod CPU instance
  ow manage start                # Managed mode (OW holds your keys)

Once running, it:
  1. Polls the job queue for pending jobs
  2. Provisions RunPod GPU workers on demand
  3. Scales up to MAX_WORKERS_PER_ORG (default 8) based on queue depth
  4. Auto-terminates idle workers after 5 minutes
  5. Cleans up unresponsive workers after 2 minutes

It's auto-scaling/serverless ONCE running — but someone must start it first.

## Normal user experience

An org admin sets up the cluster manager once (with their RUNPOD_API_KEY). After that,
regular org members only need their OPENWEIGHTS_API_KEY. They submit jobs, the cluster
manager notices, provisions GPUs, and jobs run. Users never see the RunPod key.

## What happened to me

1. First org (niels.warncke+vili@gmail.com): Freshly created, no cluster manager was
   ever set up. Jobs queued in `pending` with nothing to pick them up. I could
   connect and submit fine — but that only proved the database was reachable, not
   that compute was available.

2. I couldn't self-serve: The RUNPOD_API_KEY is not accessible from the OW org.
   Starting a cluster manager myself would have required someone to give me a
   RunPod key with billing.

3. Solution: CLR moved me to an existing org (maxime.riche.insa+spar-heuristic@gmail.com)
   that already had an active cluster manager. Jobs started immediately.

## Lesson learned

An OPENWEIGHTS_API_KEY alone is NOT sufficient to run jobs. It only talks to the
database. Compute requires either:
  (a) Someone has already set up a cluster manager for your org, OR
  (b) You have a RUNPOD_API_KEY and start the cluster manager yourself.

When setting up a new org, always confirm that a cluster manager is active before
submitting jobs.


## HuggingFace token and model storage

### How trained models get stored

When OW finishes fine-tuning, the worker pushes the model to HuggingFace. The worker
uses the org's server-side HF_TOKEN (set via `ow env import` or `ow env show`). By
default, models are pushed as PRIVATE repos under the org's HF_ORG namespace
(`push_to_private: true` in the training config).

The problem: at inference time, the OW client does a CLIENT-SIDE verification against
the HF API to check the model exists and resolve LoRA adapter details. If the model
is private, this check needs a local HF_TOKEN with read access. If you don't have one,
inference submission fails with a 404.

### The correct approach: push_to_private=False

For non-sensitive work, the simplest solution is to push models as public repos:

    job = ow.fine_tuning.create(
        model="Qwen/Qwen3-4B-Base",
        training_file=train_file_id,
        push_to_private=False,   # <-- makes inference just work
        ...
    )

No HF token needed locally. Anyone can access the model. This is the right default
for research/experiments where models aren't sensitive.

### Workaround for private models

If models must stay private (sensitive training data, proprietary work), you need the
org's HF_TOKEN locally. You can retrieve it with:

    ow env show

Then add `export HF_TOKEN=hf_...` to your local .env. This gives you read access to
the org's private HF repos so the client-side check passes.

### What happened to me

Training succeeded — workers pushed models to `longtermrisk/Qwen3-4B-Base-ftjob-*`
as private repos using the server-side HF token. My local .env had no valid HF_TOKEN,
so inference submission failed with a 404 on the client-side model check. Fixed by
retrieving the org's HF_TOKEN via `ow env show`.

In hindsight, I should have just set `push_to_private=False` since this is an open
research experiment with nothing sensitive.


## Full setup guide: running your own OW org from scratch

If I want to do fine-tuning independently (not joining someone else's org):

### 1. Create an account and org

    pip install openweights
    ow signup your@email.com
    # This gives you an OPENWEIGHTS_API_KEY
    export OPENWEIGHTS_API_KEY=ow_...

### 2. Set up HuggingFace for model storage

Create a HuggingFace account (or use an existing one). Generate a token with write
access at https://huggingface.co/settings/tokens.

For public repos (recommended for research):
    # Set server-side env so workers can push models
    # Create a .env.worker file with:
    HF_TOKEN=hf_your_token
    HF_USER=your_hf_username        # or HF_ORG=your_hf_org
    OPENWEIGHTS_API_KEY=ow_...
    RUNPOD_API_KEY=rpa_...

    # Import to OW backend
    ow env import .env.worker

    # Use push_to_private=False in training jobs — no local HF token needed

For private repos:
    # Same server-side setup as above
    # Additionally, set HF_TOKEN in your LOCAL .env so the client can verify
    # private models during inference submission
    export HF_TOKEN=hf_your_token

### 3. Get a RunPod API key

Sign up at https://runpod.io, add billing, generate an API key.
Add RUNPOD_API_KEY to your .env.worker file.

### 4. Start the cluster manager

    # Option A: run locally (simplest, must keep terminal open)
    ow cluster --env-file .env.worker

    # Option B: deploy to a cheap RunPod CPU instance (persistent, ~$0.02/hr)
    ow deploy --env-file .env.worker

    # Option C: let OW manage it (they hold your keys)
    ow env import .env.worker
    ow manage start

### 5. Submit jobs

    from openweights import OpenWeights
    ow = OpenWeights()

    file = ow.files.upload("train.jsonl", purpose="conversations")
    job = ow.fine_tuning.create(
        model="Qwen/Qwen3-4B-Base",
        training_file=file["id"],
        push_to_private=False,   # public repos for easy inference
        loss="sft",
        epochs=3,
    )

### Summary of all credentials

    Credential          Where it lives        What it does
    ─────────────────── ───────────────────── ──────────────────────────────────
    OPENWEIGHTS_API_KEY Local .env            Authenticate with OW backend
    RUNPOD_API_KEY      .env.worker (server)  Provision GPU workers
    HF_TOKEN            .env.worker (server)  Push models to HF after training
    HF_TOKEN            Local .env (optional) Access private HF repos locally
    HF_USER / HF_ORG   .env.worker (server)  Namespace for model repos on HF
